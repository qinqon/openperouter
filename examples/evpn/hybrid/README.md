# EVPN L2/L3 Stretching between On-Premises and GCP OpenShift (PoC)

This Proof of Concept (PoC) demonstrates how to extend Layer 2 and Layer 3 networks between an on-premises containerlab environment and a GCP OpenShift cluster using EVPN/VXLAN over IPsec VPN. It also showcases **KubeVirt Cross-Cluster Live Migration (CCLM)** between the two environments.

**All code for this PoC is available at:** https://github.com/qinqon/openperouter/tree/gcp

**Demo (Cross-Cluster Live Migration over L2 network):** https://asciinema.org/a/Ii6t96FdJ4w6aKfO

## Features

- **L2VNI 110**: Layer 2 stretching for VM communication between on-prem Kind and GCP (192.170.1.0/24)
- **L3VNI 100**: Inter-subnet routing via EVPN Type-5 between leafl3test (192.170.2.0/24) and GCP
- **L2VNI 666 (Migration Network)**: Dedicated stretched L2 network for KubeVirt CCLM (192.170.10.0/24)
- **Cross-Cluster Live Migration**: VMs can be live-migrated from on-prem Kind cluster to GCP OpenShift

## Terminology

**OpenPERouter CRDs** (Kubernetes custom resources):
- **Underlay**: Configures BGP peering and VTEP CIDR for the router pod
- **L2VNI**: Creates Layer 2 overlay with VXLAN, bridge interfaces, and EVPN Type-2/3 routes
- **L3VNI**: Creates VRF with VXLAN VNI for symmetric IRB inter-subnet routing

**FRR/Linux concepts** (used by leafl3test):
- **VRF**: Linux VRF for routing table isolation
- **VXLAN VNI**: VXLAN tunnel identifier for overlay traffic
- **EVPN Type-5**: IP prefix routes advertised via BGP for inter-subnet routing

**KubeVirt CCLM**:
- **Cross-Cluster Live Migration**: Live-migrate VMs between Kubernetes clusters
- **DecentralizedLiveMigration**: KubeVirt feature gate enabling CCLM
- **Migration Network**: Dedicated L2 network for VM memory transfer during migration

This script will:

##### Discovering Values from Kubernetes

Most values can be auto-discovered from Kubernetes Underlay CRDs:

```bash
# GCP VTEP CIDR (from GCP OpenShift)
kubectl --context gcp get underlay -n openperouter-system \
  -o jsonpath='{.items[0].spec.evpn.vtepcidr}'

# On-prem VTEP CIDR (from on-prem Kind)
kubectl --context prem get underlay -n openperouter-system \
  -o jsonpath='{.items[0].spec.evpn.vtepcidr}'

# Leafgcp address and ASN (from GCP Underlay neighbors with ebgpMultiHop)
kubectl --context gcp get underlay -n openperouter-system \
  -o jsonpath='{.items[0].spec.neighbors[?(@.ebgpMultiHop==true)].address}'
kubectl --context gcp get underlay -n openperouter-system \
  -o jsonpath='{.items[0].spec.neighbors[?(@.ebgpMultiHop==true)].asn}'
```

**Note**: `L3VNI_VTEP_CIDR` (leafl3test) is not discoverable from Kubernetes as it's a pure FRR container in containerlab.

The script automatically:

- Assigns whereabouts IPs to GCP instances as alias IPs
- Configures VPN gateway and tunnel with traffic selectors
- Configures Cloud Router BGP session (advertises 192.168.11.0/24 to on-prem)

VPN configuration is saved to `/tmp/gcp-vpn-config.env` for use by containerlab.

**VPN Setup Scripts:**

| Script | Description |
|--------|-------------|
| [`examples/evpn/hybrid/gcp/setup.sh`](gcp/setup.sh) | GCP-side VPN configuration (HA VPN gateway, tunnel, Cloud Router) |
| [`clab/hybrid/leafgcp/start-vpn.sh`](../../clab/hybrid/leafgcp/start-vpn.sh) | On-prem strongSwan VPN configuration (swanctl, traffic selectors, link-local BGP) |

The on-prem VPN uses **policy-based IPsec** with specific traffic selectors. The key configuration in `start-vpn.sh`:

```bash
# strongSwan swanctl.conf - Policy-based VPN
connections {
    gcp-ha-vpn {
        ...
        children {
            gcp-ha-vpn {
                # Traffic selectors define which subnets traverse the VPN
                # Use /32 for link-local to avoid bypass policy (same subnet on both sides)
                local_ts = 10.250.1.0/24,100.65.0.0/24,100.64.0.0/24,169.254.0.2/32
                remote_ts = 192.168.11.0/24,169.254.0.1/32
                ...
            }
        }
    }
}
```

**Traffic Selector Subnets:**

| Subnet | Purpose |
|--------|---------|
| `10.250.1.0/24` | Underlay BGP peering (spine ↔ leafgcp) |
| `100.65.0.0/24` | On-prem Kind cluster VTEPs (OpenPERouter) |
| `100.64.0.0/24` | On-prem leafl3test VTEPs (FRR L3VNI) |
| `169.254.0.2/32` | leafgcp link-local for Cloud Router BGP peering |
| `192.168.11.0/24` | GCP worker VTEPs |
| `169.254.0.1/32` | Cloud Router link-local BGP IP |

**Important:** Use policy-based VPN (specific traffic selectors) rather than route-based VPN (0.0.0.0/0). Route-based VPN causes xfrm template mismatch errors for forwarded traffic.

#### Prepare HCO for Cross-Cluster Live Migration

```bash
./gcp/openshift/prepare.sh
```

This patches HyperConverged to:
- Enable `DecentralizedLiveMigration` feature gate
- Configure migration network to use `migration-evpn`
- Move `virt-synchronization-controller` to worker nodes

**Note on virt-synchronization-controller placement:**

The `virt-synchronization-controller` pods need to be reachable from the source cluster during CCLM. By default they run on control plane nodes, which are not part of the EVPN overlay. There are three alternatives:

1. **Move to worker nodes** (current approach): Patch HCO to schedule `virt-synchronization-controller` on workers where the OpenPERouter overlay is available. Simple but has a downside: control plane pods have a higher priority class, making them less likely to be evicted under node resource pressure. If a `virt-synchronization-controller` pod is evicted during an active migration, the migration must restart. Running on workers increases eviction risk compared to control plane placement. This can be mitigated by configuring a custom PriorityClass for these pods.

2. **OpenPERouter node role aware multus annotation** ([PR #217](https://github.com/openperouter/openperouter/pull/217)): Implement OpenPERouter enhancement to use different multus annotation depending on nodes being control plane or workers. This removes the need to relocate the controller and preserves the default priority class behavior.

This generates per-node Underlay CRDs with mesh BGP topology:

```yaml

metadata:
  name: worker-c-xxxxx
  namespace: openperouter-system
spec:
  asn: 65001
  evpn:
    vtepcidr: 192.168.11.0/24
  nodeSelector:
    matchLabels:

```

## Cross-Cluster Live Migration

### Migration Network Configuration

Both clusters have a dedicated migration L2VNI stretched between them:

```yaml
# L2VNI 666 - Migration Network
apiVersion: openpe.openperouter.github.io/v1alpha1
kind: L2VNI
metadata:
  name: migration-net
  namespace: openperouter-system
spec:
  hostmaster:
    type: linux-bridge
    linuxBridge:
      autoCreate: true
  l2gatewayips: ["192.170.10.1/24"]
  vni: 666
  vrf: rouge
```

The migration NAD uses whereabouts with non-overlapping ranges:
- **GCP**: 192.170.10.128/25 (excludes .0-.127)
- **On-prem**: 192.170.10.0/25 (excludes .128-.255)

### Running a Migration

After setup is complete, use the `migrate.sh` script:

```bash
./migrate.sh
```

This script:
1. Creates the target VM definition on GCP with `runStrategy: WaitAsReceiver`
2. Creates VMIM receiver on GCP cluster
3. Gets the synchronization URL from the target VMIM
4. Creates VMIM sender on prem cluster pointing to the target

Manual migration steps:

```bash
# On GCP cluster - create target VM and receiver
kubectl --context gcp apply -f gcp/migrated-workload.yaml -f gcp/dst-migration.yaml

# Get the target synchronization URL
TARGET_URL=$(kubectl --context gcp get vmim vmim-target -o jsonpath='{.status.synchronizationAddresses[0]}')

# On prem cluster - initiate migration
kubectl --context prem apply -f - <<EOF
apiVersion: kubevirt.io/v1
kind: VirtualMachineInstanceMigration
metadata:
  name: vmim-source
spec:
  sendTo:
    connectURL: "${TARGET_URL}"
    migrationID: "cross-cluster-demo"
  vmiName: 192-170-1-4
EOF
```

## VNI Configurations

### L2VNI 110 (Layer 2 Stretching)

```yaml
apiVersion: openpe.openperouter.github.io/v1alpha1
kind: L2VNI
metadata:
  name: layer2
  namespace: openperouter-system
spec:
  hostmaster:
    type: linux-bridge
    linuxBridge:
      autoCreate: true
  l2gatewayips: ["192.170.1.1/24"]
  vni: 110
  vrf: red
```

- **Network**: 192.170.1.0/24
- **Gateway**: 192.170.1.1
- **VRF**: red
- **Purpose**: VM communication between on-prem and GCP

### L3VNI 100 (Inter-Subnet Routing)

```yaml
apiVersion: openpe.openperouter.github.io/v1alpha1
kind: L3VNI
metadata:
  name: red
  namespace: openperouter-system
spec:
  vrf: red
  vni: 100
```

- **VNI**: 100
- **VRF**: red (matches L2VNI)
- **Purpose**: Symmetric IRB for routing between 192.170.1.0/24 (GCP) and 192.170.2.0/24 (leafl3test)

## Key Configuration Points

1. **Traffic Selectors**: VPN must allow underlay and all VTEP subnets (configured in `gcp/network.env`):
   - `10.250.1.0/24` - Underlay BGP peering (leafgcp)
   - `100.65.0.0/24` - Kind cluster VTEPs (OpenPERouter)
   - `100.64.0.0/24` - leafl3test VTEPs (FRR L3VNI)
   - `169.254.0.x/32` - Link-local BGP peering (use /32 to avoid bypass policy)

2. **GCP Firewall**: Must allow VXLAN (UDP 4789) from all VTEP subnets

3. **GCP VTEP Reachability**: The GCP Cloud Router advertises 192.168.11.0/24 to on-prem via the BGP session with leafgcp. No static routes are needed on leafgcp - it learns the route dynamically and re-advertises it to spine

4. **BGP Topology**: On-prem uses spine as central eBGP hub (different ASNs per leaf); GCP uses eBGP mesh (different ASN per worker)

   **On-prem spine** ([`clab/hybrid/spine/frr.conf`](../../clab/hybrid/spine/frr.conf)):
   ```
   router bgp 64612
    no bgp ebgp-requires-policy
    no bgp default ipv4-unicast

    neighbor 10.250.1.1 remote-as 64512   # leafkind
    neighbor 10.250.1.3 remote-as 64515   # leafgcp
    neighbor 10.250.1.5 remote-as 64520   # leafl3test

    address-family ipv4 unicast
     neighbor 10.250.1.1 activate
     neighbor 10.250.1.3 activate
     neighbor 10.250.1.5 activate
    exit-address-family

    address-family l2vpn evpn
     neighbor 10.250.1.1 activate
     neighbor 10.250.1.3 activate
     neighbor 10.250.1.5 activate
     advertise-all-vni
    exit-address-family
   ```

   **GCP workers** (generated by [`gcp/underlay.sh`](gcp/underlay.sh)):
   ```yaml
   # Each worker gets a unique ASN and peers with all other workers + leafgcp
   spec:
     asn: 65001  # worker-1 (65002 for worker-2, etc.)
     neighbors:
       - asn: 65002
         address: 192.168.11.11    # Peer with other GCP workers
       - asn: 64515
         address: 10.250.1.3       # Peer with on-prem leafgcp
         ebgpMultiHop: true
   ```

5. **Inter-subnet routing requirements** (leafl3test ↔ GCP):
   - Same VRF name ("red") and VXLAN VNI (100) on both sides
   - leafl3test: `redistribute connected` in BGP VRF for Type-5 route advertisement
   - GCP (OpenPERouter): L3VNI CRD with matching VNI and VRF

6. **Migration Network**:
   - Dedicated L2VNI (666) stretched between clusters
   - Non-overlapping whereabouts IP ranges per cluster
   - NAD must be in KubeVirt namespace (kubevirt on Kind, openshift-cnv on OpenShift)

7. **CCLM Prerequisites**:
   - KubeVirt certificates exchanged between clusters
   - `DecentralizedLiveMigration` feature gate enabled
   - Migration network configured in KubeVirt/HCO

## Troubleshooting

### VPN Not Establishing

```bash
# Check leafgcp logs
docker logs clab-kind-leafgcp

# Check IPsec status
docker exec clab-kind-leafgcp swanctl --list-sas
docker exec clab-kind-leafgcp ipsec status
```

If VPN is not established, `swanctl --list-sas` shows empty output or CONNECTING state:

```
gcp-ha-vpn: #1, CONNECTING, IKEv2
  local  '<your-public-ip>' @ <your-public-ip>[500]
  remote '<gcp-vpn-gateway-ip>' @ <gcp-vpn-gateway-ip>[500]
```

Common issues:
- **Wrong shared secret**: Check `SHARED_SECRET` matches GCP VPN tunnel configuration
- **Firewall blocking UDP 500/4500**: Ensure NAT-T ports are open
- **Traffic selectors mismatch**: Verify local/remote traffic selectors match GCP VPN tunnel

Check xfrm errors for traffic selector issues:

```bash
docker exec clab-kind-leafgcp cat /proc/net/xfrm_stat | grep -v ": 0"
```

If you see `XfrmInTmplMismatch` errors, traffic selectors don't match:

```
XfrmInTmplMismatch: 1234
```

### BGP Sessions Not Up

```bash
# Check reachability through VPN
docker exec clab-kind-leafgcp ping -c 3 192.168.11.10

# Check FRR logs
docker exec clab-kind-leafgcp tail -f /var/log/frr/frr.log
```

Common issues:
- **ASN mismatch**: Verify `remote-as` matches the peer's actual ASN
  - Cloud Router ASN: Check with `gcloud compute routers describe <router>`
  - GCP workers: Check Underlay CRD `spec.asn`
- **No route to peer**: Verify VPN tunnel is ESTABLISHED and routes are installed
- **Firewall blocking TCP 179**: Ensure BGP port is allowed

## Productification

This section outlines the limitations of the current PoC and potential improvements to make it production-ready.

### BGP Topology

The current PoC uses a full mesh BGP topology for the underlay on GCP. This does not scale well as the number of nodes increases.

**Improvement**: Replace the underlay mesh with a Route Reflector topology, reducing BGP sessions from O(n²) to O(n).

**OpenPERouter Route Reflector Enhancement**: https://github.com/openperouter/openperouter/pull/218

### GCP IP Alias Automation

The PoC requires manual configuration of GCP IP aliases to enable VTEP-to-VTEP communication through the VPN tunnel.

**Improvement**: Develop an OpenPERouter GCP controller that automatically manages IP aliases based on cluster state.

### Alternative to ipvlan

As an alternative to ipvlan, OpenPERouter could become the primary BGP/EVPN provider (note: this would preclude using OVN-Kubernetes BGP/EVPN). Options to investigate:

- **OpenPERouter host mode with OpenShift OVN-Kubernetes**
- **Host routing or SNAT for BGP/VXLAN**
- **passt integration**

### Cross-Cluster Live Migration Controller Placement

KubeVirt's `virt-synchronization-controller` runs on control plane nodes by default, but the OpenPERouter overlay is only available on worker nodes in this PoC.

**Options**:

1. **OpenPERouter node role aware multus config enhancement** ([PR #217](https://github.com/openperouter/openperouter/pull/217)): Implement enhancement to allow router pods to use different NADs depending on whether they run on control plane or worker nodes.

2. **Relocate virt-synchronization-controller to workers**: Move the controller from control plane to worker nodes, adding placement configuration (e.g., custom PriorityClass) to prevent pod eviction.

### Static Routing Configuration

The PoC requires manual static route configuration on the GCP Cloud Router to wire up connectivity between on-prem and GCP clusters.

**Improvement**: Leverage GCP's BGP capabilities for dynamic route advertisement, eliminating manual static route management.

### VPN Traffic Selector Configuration

This PoC uses a VPN with IPSec, which requires configuring which subnets traverse the VPN tunnel (traffic selectors). This adds complexity as new networks are added. This configuration can be automated by the customer based on their network requirements.
